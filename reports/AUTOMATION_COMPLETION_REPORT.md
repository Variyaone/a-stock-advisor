### 3. 资源调度方案: reports/resource_optimization.md

详细文档（7.7KB），包含：

#### 3.1 优化策略
- **错峰执行**: 6 个时间窗口，避免资源冲突
- **限流控制**: 按优先级调度，总并发限制
- **内存管理**: 进程隔离、即时释放、内存监控

#### 3.2 资源负载曲线
```
00:00-06:00 [低负载窗口] - 健康检查、回测
06:00-09:00 [数据加载窗口] - 早盘数据更新
09:00-15:00 [交易时间] - 无任务
15:00-18:00 [数据准备窗口] - 收盘数据更新、风险检查
18:00-22:00 [推送窗口] - 每日选股推送
22:00-24:00 [静默窗口] - 无任务
```

#### 3.3 预期效果
- 启动延迟: 2-5秒 → <1秒 (↓80%)
- 内存峰值: 500MB-1GB → 50-200MB (↓75%)
- CPU 峰值: >80% → <50% (↓37.5%)
- Agent 调用: 5-10次/天 → 1-2次/天 (↓80%)

### 4. 架构文档: reports/automation_architecture.md

详细文档（17KB），包含：

#### 4.1 三层架构
1. **自动化脚本层**: 处理所有确定性任务
2. **Cron 调度层**: 负责任务的时间调度
3. **Agent 监控层**: 处理异常和需要智能决策的场景

#### 4.2 Agent 介入策略
| 场景 | 触发条件 | 处理方式 |
|-----|---------|---------|
| 异常诊断 | 任务连续失败 3 次 | Agent 分析日志 |
| 策略优化 | 回测夏普 <1.0 | Agent 调整参数 |
| 数据异常 | 质量检查失败 | Agent 重新获取 |
| 系统过载 | CPU >80% 持续 5分钟 | Agent 建议调整 |

#### 4.3 监控指标
- 系统层面: CPU、内存、磁盘、网络
- 任务层面: 成功率、执行时间、失败原因
- 业务层面: 数据延迟、推送及时性、选股准确性

   - Agent 仅在异常时介入
